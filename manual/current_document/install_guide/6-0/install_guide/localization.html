Configuring Timezone and Localization Settings

配置时区和本地化设置

描述inHybrid数据库的可用时区和本地化特性。

Parent topic: 安装和升级inHybrid

Configuring the Timezone

inHybrid数据库从一组内部存储的PostgreSQL时区中选择要使用的时区。可用的PostgreSQL时区取自Internet Assigned Numbers Authority (IANA)时区数据库，当IANA数据库为PostgreSQL更改时，inHybrid数据库根据需要更新可用时区列表。

inHybrid数据库选择匹配一个PostgreSQL时区的时区,时区服务器配置参数的值,或主机系统时区如果不设置时区。例如,在选择一个默认时区从主机系统时区,inHybrid数据库使用了一个算法来选择一个PostgreSQL时区基于主机系统时区文件。如果系统时区包含闰秒信息，inHybrid数据库就不能将系统时区与PostgreSQL时区相匹配。在本例中，inHybrid数据库根据来自主机系统的信息计算出与PostgreSQL时区的“最佳匹配”。

作为一种最佳实践，配置inHybrid数据库和主机系统以使用一个已知的、受支持的时区。这个设置时区inHybrid数据库master和segment实例,并防止inHybrid数据库选择一个最佳匹配时区每次重新启动集群,使用当前系统时区和inHybrid数据库时区文件(可能已经从IANA数据库更新自上次重启)。使用hdbconfig实用程序显示并设置inHybrid数据库时区。例如，这些命令显示inHybrid数据库时区，并将时区设置为US/Pacific。
# hdbconfig -s TimeZone
# hdbconfig -c TimeZone -v 'US/Pacific'
更改时区后必须重新启动inHybrid数据库。命令hdbstop -ra重新启动inHybrid数据库。目录视图pg_timezone_names提供inHybrid数据库时区信息。显示和设置inHybrid数据库时区的实用程序。例如，这些命令显示inHybrid数据库时区，并将时区设置为US/Pacific。

关于inHybrid数据库中的场景支持

inHybrid数据库支持本地化，有两种方法:

  • 使用操作系统的语言环境特性来提供特定于语言环境的排序顺序、数字格式等。
  • 提供inHybrid数据库服务器中定义的许多不同的字符集，包括多字节字符集，以支持以各种语言存储文本，并提供客户端和服务器之间的字符集转换。

语言环境支持是指应用程序在字母、排序、数字格式等方面尊重文化偏好。inHybrid数据库使用服务器操作系统提供的标准ISO C和POSIX语言环境设施。有关更多信息，请参考操作系统的文档。

在初始化inHybrid数据库系统时，区域设置支持会自动初始化。初始化工具hdbinitsystem将在默认情况下使用inHybrid数组的执行环境的语言环境设置来初始化这个inHybrid数组，因此，如果您的系统已经设置为在inHybrid数据库系统中使用您想要的语言环境，那么不需要做其他任何事情。

当您准备启动inHybrid数据库并且希望使用不同的语言环境(或者您不确定您的系统被设置为哪种语言环境)时，您可以通过指定-n locale选项来指示hdbinitsystem具体使用哪种语言环境。例如:

$ hdbinitsystem -c gp_init_config -n sv_SE

有关数据库初始化过程的信息，请参见初始化inHybrid数据库系统。

上面的示例将语言环境设置为瑞典语(sv)。其他的可能是en_US(美国英语)和fr_CA(加拿大)。如果一种语言环境需要多个字符集，那么规范应该是这样的:cs_CZ.ISO8859-2。在系统上的什么名称下有什么地区可用，这取决于操作系统供应商提供的内容和安装的内容。在大多数系统上，命令locale -a将提供可用locale的列表。

有时候，混合来自多个地区的规则是有用的，例如使用英语的排序规则而不是西班牙语的消息。为了支持这一点，存在一组语言环境子类别，它们只控制本地化规则的某个方面:

  • LC_COLLATE ― String sort order
  • LC_CTYPE ― Character classification (What is a letter? Its upper-case equivalent?)
  • LC_MESSAGES ― Language of messages
  • LC_MONETARY ― Formatting of currency amounts
  • LC_NUMERIC ― Formatting of numbers
  • LC_TIME ― Formatting of dates and times

如果您想让系统表现得好像没有区域设置支持一样，那么可以使用特殊的区域设置C或POSIX。

某些地区类别的性质是，它们的值在inHybrid数据库系统的生命周期内必须是固定的。也就是说，一旦hdbinitsystem运行了，就不能再更改它们了。LC_COLLATE和LC_CTYPE就是这些类别。它们影响索引的排序顺序，因此必须保持固定，否则文本列上的索引将会损坏。inHybrid数据库通过记录hdbinitsystem看到的LC_COLLATE和LC_CTYPE的值来实现这一点。服务器根据初始化时选择的语言环境自动采用这两个值。

在服务器运行时，可以根据需要更改其他语言环境类别，方法是设置与语言环境类别名称相同的服务器配置参数(有关设置服务器配置参数的更多信息，请参阅inHybrid数据库参考指南)。hdbinitsystem选择的默认值被写入到master和segment postgresql.conf配置文件中，以便在启动inHybrid数据库系统时作为默认值使用。如果你从master和每个segment postgresql.conf文件中删除这些赋值，那么服务器将从它的执行环境中继承这些设置。

请注意，服务器的语言环境行为是由服务器看到的环境变量决定的，而不是由任何客户机的环境决定的。因此，在启动系统之前，要注意在每个inHybrid数据库主机(master和segment)上配置正确的本地设置。这样做的结果是，如果客户端和服务器设置在不同的地区，消息可能出现在不同的语言中，这取决于消息的来源。

继承的地区执行环境意味着以下大多数操作系统:对于一个给定的地区类别,排序,参考以下环境变量在此订单,直到找到一个设置:LC_ALL, LC_COLLATE(变量对应于相应的类别),如果这些环境变量设置语言环境默认为C。

一些消息本地化库还查看了环境变量语言，该语言覆盖了所有其他语言环境设置，目的是设置消息的语言。如果有疑问，请参考操作系统的文档，特别是关于gettext的文档，以获得更多信息。

本地语言支持(NLS)允许将消息翻译成用户的首选语言，但inHybrid数据库中不支持英语以外的语言。这与其他语言环境支持无关。

现场行为

语言环境设置影响以下SQL特性:

  • 在查询中使用order BY对文本数据排序
  • 使用带有LIKE子句的索引的能力
  • upper, lower, and initcap 函数
  • 函数的to char系列

在inHybrid数据库中使用C或POSIX以外的语言环境的缺点是它对性能的影响。它减慢了字符处理并阻止LIKE使用普通索引。因此，只有在实际需要时才使用locale。

故障排除的地方

如果语言环境支持没有按照预期工作，请检查操作系统中的语言环境支持是否正确配置。要检查系统上安装了哪些locale，可以使用命令locale -a(如果操作系统提供的话)。

检查inHybrid数据库是否实际使用了您认为的语言环境。LC_COLLATE和LC_CTYPE设置是在初始化时确定的，如果不重新执行hdbinitsystem，就不能更改它们。其他区域设置包括LC_MESSAGES和LC_MONETARY最初由master和/或segment的操作系统环境决定，但是可以在初始化之后通过编辑每个inHybrid master和segment实例的postgresql.conf文件来更改。可以使用SHOW命令检查master主机的活动区域设置。注意，inHybrid数据库数组中的每个主机都应该使用相同的语言环境设置。

字符集的支持

设置的字符集

hdbinitsystem通过在初始化时读取gp_init_config文件中编码参数的设置来定义inHybrid数据库系统的默认字符集。默认字符集是UNICODE或UTF8。

除了系统默认设置之外，还可以用不同的字符集创建数据库。例如:

=> CREATE DATABASE korean WITH ENCODING 'EUC_KR';
重要提示:尽管可以指定数据库所需的任何编码，但选择与所选语言环境不一致的编码是不明智的。LC_COLLATE和LC_CTYPE设置暗示了一种特殊的编码，依赖于语言环境的操作(例如排序)很可能会误解不兼容编码的数据。

由于hdbinitsystem冻结了这些语言环境设置，因此在不同的数据库中使用不同编码的灵活性只是理论上的，而不是实际的。

安全使用多种编码的一种方法是在初始化期间将语言环境设置为C或POSIX，从而禁用任何真正的语言环境感知。

要启用自动字符集转换，您必须告诉inHybrid数据库您希望在客户机中使用的字符集(编码)。有几种方法可以做到这一点:

  • 使用psql中的\encoding命令，它允许您动态更改客户机编码。
  • 使用SET client_encoding TO。设置客户端编码可以用这个SQL命令: 
    => SET CLIENT_ENCODING TO 'value';

    查询当前客户端编码: 

    => SHOW client_encoding;

    返回默认编码: 

    => RESET client_encoding;
  • 使用PGCLIENTENCODING环境变量。当在客户端环境中定义PGCLIENTENCODING时，在与服务器建立连接时将自动选择该客户端编码。(随后可以使用上面提到的任何其他方法来覆盖它。)
  • 设置配置参数client_encoding。如果在主postgresql.conf文件中设置了client_encoding，那么在连接到inHybrid数据库时将自动选择该客户机编码。(随后可以使用上面提到的任何其他方法来覆盖它。)

如果无法转换特定字符―假设您选择EUC_JP作为服务器，拉丁1作为客户端，那么一些日语字符没有LATIN1的表示―然后报告一个错误。

如果客户端字符集定义为SQL_ASCII，则无论服务器的字符集如何，编码转换都将被禁用。除非处理的是全ASCII数据，否则使用SQL_ASCII是不明智的。不支持SQL_ASCII作为服务器编码。

不是所有的api都支持所有列出的字符集。例如，JDBC驱动程序不支持MULE_INTERNAL、LATIN6、LATIN8和LATIN10。
SQL_ASCII设置的行为与其他设置有很大的不同。字节值0-127根据ASCII标准解释，字节值128-255作为未解释字符。如果您正在处理任何非ASCII数据，那么将SQL_ASCII设置用作客户端编码是不明智的。不支持SQL_ASCII作为服务器编码。
